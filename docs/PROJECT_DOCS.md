# Документация проекта SecureGuard

## 1. Назначение проекта

`SecureGuard` — это Windows-first desktop-приложение для хранения паролей с локальным шифрованием записей, Tauri-клиентом на `Rust + HTML/CSS/JS`, gRPC-бэкендом на `Go` и хранилищем на `PostgreSQL`.

Проект находится в стадии MVP. Основной пользовательский сценарий уже реализован: регистрация, вход, создание защищённых записей, просмотр списка, расшифровка записи по сид-фразе и удаление.

## 2. Цели проекта

- Дать пользователю локальное десктопное хранилище паролей.
- Выполнять шифрование записей до отправки на сервер.
- Хранить пользовательские предпочтения и записи в централизованном бэкенде.
- Предоставить базовые защитные механизмы на Windows:
  - блокировку скриншотов окна;
  - контроль автозапуска;
  - запуск с повышенными правами;
  - антиотладочные механизмы в release-сборке.
- Обеспечить основу для дальнейшего развития:
  - аналитика;
  - административные функции;
  - расширение API;
  - усиление security-модели.

## 3. Текущий функционал

### 3.1. Пользовательские функции

- Регистрация нового аккаунта.
- Авторизация по логину и паролю.
- Хранение списка записей с паролями.
- Создание новой записи.
- Удаление записи.
- Расшифровка записи по сид-фразе и копирование в буфер обмена.
- Автоматическая очистка буфера обмена через 30 секунд.
- Смена языка интерфейса: `RU/EN`.
- Выбор алгоритма шифрования:
  - `AES-256-GCM + Argon2id`;
  - `AES-256-GCM + SHA-256`.
- Переключение темы.
- Настройка автоблокировки.
- Настройка подтверждения удаления.
- Блокировка контекстного меню в приложении.
- Настройка запуска вместе с Windows.
- Включение и отключение `Screenshot Guard`.

### 3.2. Административные функции

- Получение суммарной статистики по системе.
- Получение статистики за текущий день.
- Отображение графиков активности и регистраций.
- Получение топа сервисов.
- Получение распределения по криптоалгоритмам.

Административная статистика доступна через бэкенд и используется клиентом для staff/admin-сценариев.

## 6. Пользовательские сценарии

### 6.1. Регистрация

1. Пользователь вводит логин, пароль и сид-фразу.
2. Клиент вызывает `LoginService.Register`.
3. Бэкенд:
   - проверяет входные данные;
   - хэширует пароль аккаунта через `bcrypt`;
   - создаёт пользователя;
   - создаёт preferences;
   - открывает сессию.
4. Клиент сбрасывает локальную сессию и переводит пользователя на экран входа.

### 6.2. Вход

1. Пользователь вводит логин и пароль.
2. Клиент вызывает `LoginService.Authorize`.
3. Бэкенд проверяет пароль и создаёт/возвращает сессию.
4. Клиент сохраняет сессионный идентификатор в памяти процесса и помечает пользователя как authenticated.

### 6.3. Добавление записи

1. Пользователь указывает название записи, пароль и сид-фразу.
2. Клиент локально шифрует пароль.
3. На сервер отправляются:
   - зашифрованный пароль;
   - salt;
   - метаданные записи.
4. Бэкенд сохраняет запись в БД.

### 6.4. Просмотр и копирование пароля

1. Клиент получает список записей.
2. Для просмотра/копирования пользователь вводит сид-фразу.
3. Расшифровка выполняется локально в клиенте.
4. Расшифрованный пароль помещается в буфер обмена.
5. Через 30 секунд буфер обмена очищается автоматически.

### 6.5. Управление настройками

Пользователь может менять:

- язык интерфейса;
- тему;
- алгоритм шифрования;
- режим `Screenshot Guard`;
- автозапуск;
- подтверждение удаления;
- автоблокировку;
- блокировку контекстного меню.

Часть настроек синхронизируется с сервером, часть хранится локально в настройках клиента.

## 8. Схема данных PostgreSQL

### 8.1. `users`

Содержит:

- `id`
- `username`
- `password`
- `seed_phrase`
- `admin_access`
- `joined`

Назначение:

- учётные записи пользователей.

### 8.2. `preferences`

Содержит:

- `owner`
- `theme`
- `lang`
- `crypt`

Назначение:

- пользовательские настройки.

### 8.3. `passwords`

Содержит:

- `id`
- `owner`
- `service`
- `login`
- `pass`
- `salt`
- `created_at`

Назначение:

- защищённые записи пользователя.

### 8.4. `sessions`

Содержит:

- `id`
- `owner`
- `client_hash`
- `revoked`
- `created`
- `expires`

Назначение:

- активные и завершённые сессии.

Особенность:

- TTL сессии по схеме БД составляет `90 минут`.
- Фоновый воркер каждые `30 секунд` ищет просроченные сессии и отзывает их.

### 8.5. `statistics`

Содержит агрегированную дневную статистику:

- latency percentiles;
- top services;
- crypto usage;
- timestamp.

### 8.6. `activity`

Содержит часовую статистику активности:

- число пользователей;
- число регистраций;
- timestamp.

## 9. Фоновые процессы

### 9.1. Session cleanup worker

- запускается при старте бэкенда;
- каждые 30 секунд проверяет истекшие сессии;
- отзывает найденные сессии.

### 9.2. Statistics persistence worker

Запускает два фоновых цикла:

- почасовое сохранение активности;
- ежедневное сохранение статистики.

## 10. Модель безопасности

### 10.1. Что уже реализовано

- Локальное шифрование записей перед отправкой на сервер.
- Поддержка двух алгоритмов шифрования:
  - `AES-256-GCM + Argon2id`;
  - `AES-256-GCM + SHA-256`.
- Хэширование пароля аккаунта на сервере через `bcrypt`.
- Сессионная аутентификация с привязкой к client metadata.
- Очистка буфера обмена через 30 секунд.
- `Screenshot Guard` на Windows.
- Автозапуск через реестр Windows.
- Запуск приложения с повышением привилегий на Windows.
- Антиотладочные механизмы в release-сборке Windows.

### 10.2. Важные ограничения текущего MVP

Это нужно фиксировать в любой серьёзной документации или защите проекта:

- Проект не является zero-knowledge-хранилищем в строгом смысле.
- Сид-фраза аккаунта при регистрации передаётся на сервер и в текущей реализации хранится в БД.
- Сид-фраза присутствует в серверной пользовательской модели и требует пересмотра перед production.
- gRPC-сервис в локальном режиме обычно работает без TLS внутри dev-среды.
- Антиотладочные и anti-tamper механизмы сильно завязаны на Windows и release-режим.
- Клиентский UX уже покрывает core flow, но часть серверного API ещё не выведена в полноценные пользовательские экраны.

### 10.3. Что стоит усилить перед production

- убрать хранение сид-фразы аккаунта в открытом виде;
- формализовать ключевую модель и recovery flow;
- ввести полноценную transport security-конфигурацию;
- провести аудит антиотладочной логики и прав администратора;
- разделить security-функции по профилям запуска;
- расширить тестирование криптографии и e2e-сценариев.

## 11. Запуск проекта

### 11.1. Быстрый локальный запуск под Windows

Основной путь:

```bat
run.bat
```

Скрипт умеет:

- остановить старые процессы;
- проверить backend;
- создать `server/starter/.env`, если файла нет;
- поднять PostgreSQL в Docker;
- применить SQL-схему;
- запустить Go-бэкенд;
- запустить Tauri-клиент в `dev` или `build`.

### 11.2. Ручной запуск

### Backend

```bash
cd server/starter
go run .
```

### Frontend

```bash
cd client
npm run dev
```

### База данных

Можно использовать Docker:

```bash
docker run --name secureguard-postgres -e POSTGRES_USER=postgres -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=secureguard -p 5432:5432 -d postgres:16-alpine
```

### 11.3. Docker Compose

```bash
docker compose up --build
```

Поднимаются:

- `db`
- `backend`
- `caddy`

## 12. Конфигурация

Ключевые переменные окружения бэкенда:

- `POSTGRES_HOST`
- `POSTGRES_PORT`
- `POSTGRES_USER`
- `POSTGRES_PASSWORD`
- `POSTGRES_NAME`
- `POSTGRES_TLS`
- `BOOT_PORT`
- `LOG_SERVICE`
- `LOG_LEVEL`
- `KAFKA_ENABLED`
- `KAFKA_BROKERS`
- `KAFKA_TOPIC`
- `KAFKA_CLIENT_ID`
- `DEBUG_MODE`

Ключевые переменные клиента:

- `SECUREGUARD_GRPC_ENDPOINT`
- `SECUREGUARD_BACKEND`

Важно:

- в коде бэкенда дефолтный `BOOT_PORT` равен `50051`;
- для локального Tauri-клиента в проекте используется совместимый сценарий с `8080`;
- `run.bat` создаёт `.env` c `BOOT_PORT=8080`, чтобы клиент подключался без дополнительной настройки.
