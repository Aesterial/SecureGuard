# Документация проекта SecureGuard

## 1. Назначение и текущее состояние

`SecureGuard` — это Windows-first desktop-приложение для хранения паролей. Решение состоит из:

- Tauri-клиента на `Rust + HTML/CSS/JS`;
- gRPC-бэкенда на `Go`;
- `PostgreSQL` для постоянного хранения;
- protobuf-контрактов в `api/`.

По состоянию текущего репозитория проект находится на стадии рабочего MVP. Основной пользовательский сценарий реализован и проверяется кодом:

- регистрация;
- вход;
- создание записи;
- просмотр списка записей;
- локальная расшифровка записи по сид-фразе;
- копирование пароля в буфер обмена;
- удаление записи;
- локальные и частично серверные настройки;
- staff-only экран административной статистики.

## 2. Состав решения

### 2.1. `client/`

`client/src/` содержит интерфейс приложения:

- страницы входа и регистрации;
- экран хранилища;
- экран настроек;
- staff-only экран статистики;
- локальное состояние UI и настройки.

`client/src-tauri/src/` содержит нативную часть:

- `main.rs` — Tauri-команды и состояние приложения;
- `api.rs` — gRPC-клиент и маппинг protobuf-ответов;
- `crypto.rs` — локальная криптография;
- `screenshot_guard.rs` — Windows-специфичная защита окна;
- `protection.rs` — release-only антиотладочные и anti-tamper механизмы.

### 2.2. `server/`

`server/starter/`:

- загружает конфиг;
- поднимает gRPC-сервер;
- регистрирует сервисы;
- запускает фоновые воркеры.

`server/internal/` разделён на слои:

- `domain/` — сущности и контракты;
- `app/` — бизнес-логика;
- `infra/` — gRPC, БД, логирование, rate limiting.

### 2.3. `api/`

Содержит protobuf-контракты и используется для генерации:

- Go-stub'ов в `server/internal/api/...`;
- Rust-stub'ов в `client/grpc/...`.

## 3. Реально реализованный функционал

### 3.1. Пользовательский desktop-flow

В текущем UI реально доступны:

- регистрация нового аккаунта;
- вход по логину и паролю;
- локальный выбор алгоритма шифрования:
  - `AES-256-GCM + Argon2id`;
  - `AES-256-GCM + SHA-256`;
- создание записи;
- просмотр списка записей;
- локальная расшифровка и копирование пароля по сид-фразе;
- удаление записи;
- RU/EN интерфейс;
- светлая/тёмная тема;
- `Screenshot Guard`;
- запуск с Windows;
- авто-выход по таймеру бездействия;
- таймаут очистки буфера обмена;
- подтверждение удаления;
- блокировка контекстного меню.

Важно:

- текущий UI работает именно с названием записи и самим секретом;
- отдельного поля "логин аккаунта" на экране создания записи сейчас нет;
- изменение существующей записи через UI не реализовано.

### 3.2. Staff/admin-функции

В клиенте есть staff-only экран статистики с read-only доступом:

- суммарные счётчики пользователей / администраторов / паролей / активных сессий;
- график активности;
- график регистраций;
- top services;
- распределение по криптоалгоритмам.

Управление пользователями из UI не реализовано.

### 3.3. Backend-only и частично задействованные возможности

В репозитории есть функциональность, которая существует на backend/Tauri-уровне, но не выведена в полноценный пользовательский сценарий:

- `PasswordService.Update`;
- `SessionsService.GetList` и `SessionsService.Revoke`;
- `UserService.ChangeKey`;
- Tauri-команды `store_vault_envelope`, `get_vault_envelope`, `rotate_seed_phrase`.

Отдельно:

- `UserService.List` объявлен в protobuf;
- серверный `UserService` не реализует этот RPC и использует generated `Unimplemented` fallback.

## 4. Архитектура и поток данных

### 4.1. Логическая схема

```text
Пользователь
  -> UI (HTML/CSS/JS)
  -> Tauri / Rust commands
  -> gRPC API
  -> Go backend
  -> PostgreSQL
```

### 4.2. Регистрация

Текущая реализация отличается от старой документации:

1. Пользователь вводит логин, пароль и сид-фразу.
2. Клиент локально формирует `master key envelope`.
3. В `LoginService.Register` отправляются:
   - логин;
   - пароль;
   - `wrapped master key`;
   - `salt`;
   - параметры KDF.
4. Сервер:
   - хэширует пароль через `bcrypt`;
   - создаёт пользователя;
   - создаёт `preferences`;
   - сохраняет данные ключа в `users_keys`;
   - открывает сессию.
5. После регистрации клиент принудительно очищает локальную сессию и переводит пользователя обратно на экран входа.

Критично:

- сырая сид-фраза на сервер не отправляется;
- сервер хранит не её, а обёрнутый мастер-ключ и KDF-параметры.

### 4.3. Вход

1. Клиент вызывает `LoginService.Authorize`.
2. Сервер проверяет пароль и создаёт новую сессию.
3. Клиент хранит session token в памяти процесса.
4. Все дальнейшие gRPC-запросы отправляются с metadata:
   - `session`;
   - `client`.

Сервер:

- сверяет токен с сессией;
- привязывает её к `client_hash`;
- обновляет `last_seen`.

### 4.4. Создание записи

1. Пользователь вводит название, пароль и сид-фразу.
2. Клиент:
   - либо разворачивает уже известный мастер-ключ;
   - либо генерирует новый;
   - шифрует пароль локально;
   - заново оборачивает мастер-ключ сид-фразой.
3. На сервер уходят:
   - ciphertext;
   - JSON-метаданные записи;
   - `wrapped_master_key`;
   - `salt`;
   - алгоритм шифрования.
4. Сервер сохраняет запись в таблицу `passwords`.

### 4.5. Просмотр и копирование

1. Клиент получает список записей.
2. Пользователь вводит сид-фразу.
3. Клиент локально:
   - разворачивает мастер-ключ;
   - расшифровывает пароль;
   - копирует его в буфер обмена.
4. Через заданный таймаут буфер очищается.

По умолчанию UI использует `30 секунд`, но сама Tauri-команда принимает диапазон `5..300` секунд.

## 5. Актуальная gRPC-поверхность

### 5.1. `LoginService`

Реально реализовано на сервере:

- `Register`
- `Authorize`
- `Logout`

Используется текущим desktop-клиентом полностью.

### 5.2. `PasswordService`

Реально реализовано на сервере:

- `List`
- `Create`
- `Update`
- `Delete`

Текущий desktop-клиент использует:

- `List`
- `Create`
- `Delete`

`Update` остаётся backend-only.

### 5.3. `UserService`

В protobuf объявлены:

- `Info`
- `List`
- `ChangeTheme`
- `ChangeLanguage`
- `ChangeCrypt`
- `ChangeKey`

Фактически реализованы сервером:

- `Info`
- `ChangeTheme`
- `ChangeLanguage`
- `ChangeCrypt`
- `ChangeKey`

Не реализован прикладным серверным кодом:

- `List`

Desktop-клиент сейчас использует только:

- `ChangeTheme`
- `ChangeLanguage`
- `ChangeCrypt`

### 5.4. `StatsService`

Реально реализовано:

- `Today`
- `ByDate`
- `Total`

Desktop-клиент использует:

- `Today`
- `Total`

`ByDate` доступен на backend-уровне, но не выведен в отдельный экран или фильтр по дате.

### 5.5. `SessionsService`

Реально реализовано:

- `GetList`
- `Revoke`

Сервис регистрируется в `server/starter/start.go`, но текущий desktop-клиент его не вызывает.

## 6. Схема данных PostgreSQL

### 6.1. `users`

Реальные поля:

- `id`
- `username`
- `password`
- `admin_access`
- `joined`

Назначение:

- учётные записи;
- роль staff/admin определяется через `admin_access`.

### 6.2. `preferences`

Реальные поля:

- `owner`
- `theme`
- `lang`
- `crypt`

Назначение:

- серверные пользовательские предпочтения.

### 6.3. `users_keys`

Реальные поля:

- `owner`
- `master_key`
- `salt`
- `version`
- `memory`
- `iterations`
- `parallelism`

Назначение:

- хранение `wrapped master key`;
- хранение параметров KDF.

Это ключевое отличие от старого описания проекта: таблицы с сырой сид-фразой в текущей схеме нет.

### 6.4. `passwords`

Реальные поля:

- `id`
- `owner`
- `service`
- `login`
- `ciphertext`
- `version`
- `nonce`
- `aad`
- `metadata`
- `created_at`

Фактическое использование текущим клиентом:

- `service` хранит заголовок/название записи;
- `ciphertext` хранит зашифрованный секрет;
- `login` и часть служебных данных используются как контейнер для JSON-метаданных;
- `metadata` хранит формат записи.

То есть текущая модель уже не совпадает с простым описанием вида "service/login/pass/salt".

### 6.5. `sessions`

Реальные поля:

- `id`
- `owner`
- `client_hash`
- `revoked`
- `revoked_at`
- `created`
- `expires`
- `last_seen`

Особенности:

- TTL по схеме БД: `90 минут`;
- cleanup worker проверяет истёкшие сессии каждые `30 секунд`.

### 6.6. `statistics`

Содержит дневные снимки:

- `p50`
- `p90`
- `services_top`
- `crypt_uses`
- `at`

### 6.7. `activity`

Содержит почасовые снимки:

- `users`
- `registers`
- `at`

## 7. Настройки и их фактическое хранение

### 7.1. Что хранится локально

Текущий UI гарантированно хранит локально:

- `Screenshot Guard`;
- `startup with Windows`;
- авто-выход;
- таймаут буфера обмена;
- подтверждение удаления;
- блокировку контекстного меню;
- локально сохранённый `master key envelope`;
- локальный выбранный алгоритм шифрования.

### 7.2. Что умеет синхронизироваться с сервером

Через `UserService` backend умеет сохранять:

- тему;
- язык;
- криптоалгоритм.

### 7.3. Важный нюанс по hydration настроек

По текущей реализации:

- тема и язык корректно приходят в `UserSelf.preferences` и применяются в UI после входа;
- API для `ChangeCrypt` реализован и значение сохраняется в БД;
- при этом desktop-клиент всё равно опирается на локальные настройки при старте, потому что восстановление crypto-preference из серверного payload неполное.

Иными словами:

- серверная запись crypto-preference существует;
- клиентская bootstrap-логика всё ещё частично локальная.

## 8. Инфраструктурные режимы запуска

### 8.1. Минимальный локальный режим

Минимум для core vault flow:

- `PostgreSQL`;
- backend;
- desktop client.

Именно под этот сценарий `run.bat` создаёт упрощённый `.env` с `KAFKA_ENABLED=false`.

Следствие:

- основной password-vault flow работает;
- staff analytics в таком режиме не работает.

### 8.2. Полный Docker stack

`docker-compose.yml` поднимает:

- `db`
- `redis`
- `kafka`
- `backend`
- `caddy`

Этот режим нужен для:

- логирования через Kafka;
- чтения логов для статистики;
- Redis-based rate limiting;
- полного server-side окружения.

## 9. Фоновые процессы

### 9.1. Session cleanup worker

- стартует вместе с backend;
- раз в `30 секунд` получает список истёкших сессий;
- отзывает их.

### 9.2. Statistics persistence worker

Стартует двумя независимыми циклами:

- почасовое сохранение `activity`;
- ежедневное сохранение `statistics`.

Важно:

- оба сценария статистики завязаны на log reader;
- без Kafka/log reader статистический pipeline не даёт полноценного результата.

## 10. Модель безопасности и ограничения

### 10.1. Что реально реализовано

- локальное шифрование секретов до отправки на backend;
- поддержка двух алгоритмов derivation/encryption для сид-фразы;
- `bcrypt` для пароля аккаунта;
- session-based auth;
- привязка сессии к `client_hash`;
- очистка буфера обмена по таймеру;
- Windows screenshot protection;
- Windows startup через реестр;
- release-only anti-debug / anti-tamper код в клиенте.

### 10.2. Что важно понимать корректно

- проект не является строгим zero-knowledge-хранилищем;
- сервер не хранит сырую сид-фразу, но хранит `wrapped master key`;
- записи в БД уже содержат зашифрованный материал и служебные метаданные;
- debug/local режим backend обычно работает без TLS;
- release-клиент ожидает `https://` endpoint;
- `Screenshot Guard` и startup-интеграция ориентированы на Windows;
- значительная часть защитной логики включается только в Windows release-сборке.

### 10.3. Что пока не выведено в полноценный продуктовый UX

- изменение существующей записи;
- управление активными сессиями;
- ротация ключа/сид-фразы из UI;
- user management;
- полноценная работа admin analytics в "минимальном" локальном режиме без Kafka.

## 11. Запуск, сборка и генерация

### 11.1. Быстрый запуск под Windows

Основной путь:

```bat
run.bat
```

Скрипт:

- проверяет backend;
- создаёт `server/starter/.env`, если файла нет;
- поднимает `PostgreSQL` в Docker при необходимости;
- применяет схему;
- запускает backend;
- запускает Tauri-клиент.

### 11.2. Ручной запуск

Core flow:

```bash
docker run --name secureguard-postgres -e POSTGRES_USER=postgres -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=secureguard -p 5432:5432 -d postgres:16-alpine
docker exec -i secureguard-postgres psql -U postgres -d secureguard < server/migrations/scheme/sqlc.sql

cd server/starter
go run .

cd client
npm run dev
```

Полный stack:

```bash
docker compose up --build
```

### 11.3. Генерация protobuf-кода

```bash
cd api
buf generate
```

Результат:

- Go stubs -> `server/internal/api/...`
- Rust stubs -> `client/grpc/...`

## 12. Верификация по репозиторию

При приведении документации в порядок были повторно проверены базовые инженерные проверки:

- `go test ./...` в `server/internal` — проходит;
- `go test ./...` в `server/starter` — проходит;
- `cargo check` в `client/src-tauri` — проходит.

Замечание:

- у Rust-клиента есть предупреждения по неиспользуемому generated/protection-коду, но компиляция успешна;
- для Go в текущей среде понадобился локальный `GOCACHE`, чтобы обойти ошибку доступа к глобальному cache trim.

## 13. Итоговая формулировка проекта

Коротко и корректно по текущему коду:

> SecureGuard — это настольное приложение для хранения паролей, в котором секреты шифруются на клиенте, передаются в Go-бэкенд по gRPC и сохраняются в PostgreSQL. Текущий MVP покрывает основной vault-flow, локальные защитные механизмы Windows, серверные пользовательские настройки и staff-only статистику, но часть API остаётся backend-only и не доведена до полноценного UI/production hardening.
